1. Embedding Type Inconsistencies
MAJOR DISCREPANCY: Different default embedding types across modules:

Vector Search Module:
Default: 'customer_profile'
Alternative: 'profile_demographic' (in segmentation)
Usage: Consistent in search validation and queries
Embedding Generation Module:
Enhanced Service: 'customer_profile'
Queue Service: 'profile_demographic'
Analytics Routes: 'profile_demographic'
Legacy Routes: 'profile_summary'
Impact: Search operations may fail to find embeddings generated by different services due to type mismatch.

2. Data Column Usage Discrepancy
CRITICAL ISSUE: Mixed usage of legacy vs optimized vector columns:

Vector Search:
-- Uses legacy 'embedding' column (real[])
SELECT embedding FROM customer_embeddings
WHERE customer_id = $1 AND embedding_type = $2
Database Schema:
embedding: real("embedding").array().notNull(), // DEPRECATED
embeddingVector: vector("embedding_vector"), // NEW: Optimized pgvector
Problem: Vector search still uses deprecated real[] column instead of optimized vector(1536) column.

3. API Parameter Name Inconsistencies
Vector Search API:
Parameter: embeddingType (camelCase)
Database column: embedding_type (snake_case)
SQL queries: embedding_type
Embedding Generation:
Parameter: embeddingType (camelCase)
Database column: embedding_type (snake_case)
Sometimes hardcoded as string literals
Result: Inconsistent parameter mapping between API and database.

4. Data Format Handling Differences
Vector Search Module:
// Expects pre-parsed embedding arrays
const embeddingVector = `[${embedding.join(',')}]`;
Embedding Generation:
// Stores as number array directly
embedding: embedding, // Number array format for PostgreSQL real array
Issue: Different expectations for embedding data format between reading and writing operations.

5. Error Handling Inconsistencies
Vector Search:
Uses secureLogger with structured metadata
Throws Error objects with descriptive messages
Has comprehensive validation with security logging
Embedding Generation:
Uses applicationLogger with different format
Mixed error handling approaches across services
Some services use ExternalApiError, others use generic Error
6. Validation Schema Differences
Vector Search Validation:
embeddingType: z.string().optional().default('customer_profile')
Storage Interface:
upsertCustomerEmbedding(customerId: string, embedding: number[], embeddingType: string)
Inconsistency: Optional vs required embedding type handling.

7. Performance Optimization Gaps
Vector Search:
Uses HNSW indexing
Native pgvector operations with <=> operator
Sub-second performance targets
Embedding Generation:
Still writes to legacy real[] column
No optimization for vector column usage
Missing integration with HNSW index creation
